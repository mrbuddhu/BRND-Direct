Supabase credentials stay secret inside the Worker — never exposed to the browser
Why Cloudflare Workers? Your portal is a static site (no server). The Cloudflare Worker acts as a secure API proxy — it holds your Supabase keys as encrypted secrets and makes the database calls server-side. Your buyer's browser only talks to /api/products, never directly to Supabase.
Current State: Your buyer/products.html uses hardcoded static product data (22 sample products). After completing this guide, it will load your full live catalog from Supabase in real-time with search, pagination, and inventory counts.

2
Enable Supabase Row Level Security for Products
Allow the Worker to read products and brands tables

Since the Worker uses the service_role key, it bypasses Row Level Security (RLS) automatically. However you still need to make sure RLS is set correctly so that:
• The Worker (service_role) can read ALL products
• If you want buyers to query products directly (future), RLS policies are in place

Run this SQL in Supabase SQL Editor
Open Supabase → SQL Editor → New Query → paste and run:
sql 
-- Enable RLS (already done if you ran schema.sql)
ALTER TABLE products ENABLE ROW LEVEL SECURITY;
ALTER TABLE brands   ENABLE ROW LEVEL SECURITY;

-- Allow public read access to active products
-- (The Worker also reads these via service_role, bypassing RLS)
CREATE POLICY "Public can read active products"
  ON products FOR SELECT
  USING (is_active = true);

CREATE POLICY "Public can read active brands"
  ON brands FOR SELECT
  USING (is_active = true);

-- Verify tables have data
SELECT COUNT(*) FROM products;
SELECT COUNT(*) FROM brands;
If the COUNT query returns 0, your catalog is empty. You need to add products to Supabase first — either through the Supabase Table Editor UI, or by running INSERT statements. See the DB Schema Map section below for the exact column names.

Add Secrets to Cloudflare Worker
Store your Supabase credentials securely — two methods available
Method A — Cloudflare Dashboard (Recommended — No Terminal)
This is the easiest method. No command-line needed.
1
Open Cloudflare Dashboard
Go to dash.cloudflare.com → click Workers & Pages in the left sidebar.
2
Open your Worker
Click your worker named brnd-direct-api (or the name shown in your Cloudflare dashboard).
3
Go to Settings → Variables & Secrets
Click the Settings tab → then Variables and Secrets section → click "Add variable".
4
Add the two secrets
Add each one as a Secret (not plain variable) so they're encrypted:
Variable NameTypeValue
SUPABASE_URLSecretYour Project URL from Step 1
SUPABASE_SERVICE_ROLE_KEYSecretYour service_role JWT from Step 1
Click Deploy after saving each secret. The secrets are now encrypted and injected into your Worker at runtime.
Method B — Wrangler CLI (Advanced)
If you prefer the command line:
bash 
# 1. Install Wrangler globally (one-time)
npm install -g wrangler

# 2. Login to Cloudflare
wrangler login

# 3. Set Supabase URL (you'll be prompted to paste the value)
wrangler secret put SUPABASE_URL

# 4. Set Supabase service role key
wrangler secret put SUPABASE_SERVICE_ROLE_KEY

# 5. Verify secrets are set
wrangler secret list

Troubleshooting
Common issues and how to fix them
Products page still shows static/demo data
  • Check browser console (F12) — is there an API error?
  • Visit /api/products directly — do you get JSON back?
  • Verify SUPABASE_URL and SUPABASE_SERVICE_ROLE_KEY are set in CF Dashboard
  • Redeploy the Worker after adding secrets
  • Make sure Supabase products table has rows (SELECT COUNT(*) FROM products)
CORS Error in browser console
  • The Worker already sets Access-Control-Allow-Origin: * — this should not happen
  • If your portal is on a custom domain, check the _headers file in your project root
  • Try opening the Worker URL directly — if it returns JSON, CORS is fine and the issue is elsewhere
401 / 403 Unauthorized from Supabase
  • You are using the service_role key, not the anon key — double-check
  • The service_role key starts with eyJ and is longer than the anon key
  • In Supabase Settings → API, there are two keys: anon (public) and service_role (secret). Use service_role.
Products load but show wrong fields / empty names
  • Open Supabase → Table Editor → products — check your columns match the schema
  • Make sure brand_id in products table points to a valid row in brands table
  • The API joins brands — if brand is null, product card shows "Unknown Brand"
Worker not found / 404 on /api/products
  • In Cloudflare Dashboard → Workers & Pages → your project → Settings → Functions → Routes — check that /api/* routes to your Worker
  • Or set it up as a Pages Function by placing the file in functions/api/ directory
  • Make sure the Worker was actually deployed — click "Edit Code" and confirm the products routes are visible
